### **第 1 层：短期记忆（STM）**

**类比**：人在当下对话里能记住的事，类似"工作记忆中的当前场景"

**作用**：用滑动窗口维护最近 N 条（当前上限 1000）对话。Agent 每次回复前，先把这些消息喂给大模型作为对话上下文，保证它不会"前言不搭后语"。

**为什么必须有**：大模型本身无状态，每次 API 调用都是全新的——如果不主动把历史消息传进去，它根本不知道"上面刚说了什么"。STM 就是解决这个最基础的"失忆"问题。

**关键机制**：历史条数到达上限时触发 `rebuild_history()`，把最旧的一批"卸载"出去——这个卸载动作是情景记忆和决策记忆的写入触发点。

---

### **第 2 层：情景记忆（Episodic）**

**类比**："上次开会聊了什么"——按时间顺序索引的历史事件

**作用**：把"卸载"出来的历史消息用 LLM 提炼成结构化知识片段（`fact / preference / goal / task` 等类型），插入 **LightRAG 图数据库**（向量 + 知识图谱双存储）。下次用户提问时，通过语义检索把相关历史拿回来。

**为什么必须有**：STM 是有限窗口，超过 1000 条的内容就丢了。情景记忆是 STM 的"压缩备份"——代价是损失细节，换取跨会话的长期检索能力。

**关键区别（对比 STM）**：STM 是全文、按顺序，情景记忆是提炼后的事实点、按语义检索。

---

### **第 3 层：语义记忆（Semantic）**

**类比**："公司组织架构" / "产品核心卖点"——不带时间感的永久性事实

**作用**：存储不依附于任何一次具体对话的领域知识（如文档库、知识库、SOP 规范）。检索时按概念关系而非时间顺序。

**为什么有但没实现**：这层需要对接企业 Wiki、知识库等外部数据源，需要额外接入工程，当前 9 天时间窗口内不是重点，先占位。本项目的 LightRAG 图实际上部分承担了这个角色。

---

### **第 4 层：工作记忆（Working）**

**类比**：做多步任务时的"便签纸"——跨轮次暂存中间结果

**作用**：当 Agent 在处理"起草报告""多表单提取"等需要多步骤的任务时，暂存中间状态（键值对），任务完成后自动清理。

**为什么有但没实现**：适用于长流程自动化场景。当前 Demo 聚焦飞书群聊的消息-响应模式，Working Memory 价值体现不充分，暂列为后续扩展。

---

### **第 5 层：决策记忆（Decision Layer）—— 本项目核心新增**

**类比**：团队会议的"正式档案柜"——存决策结论、理由、反对意见、时效

**作用**：从群聊/文档/CLI 操作中自动抽取"决策三元组（Decision-Reason-Conclusion）"，存入 SQLite（结构化事实）+ LightRAG 图（语义关系），并附带：

- **遗忘曲线**（艾宾浩斯 `R = e^(-Δt/S)`）：自动衰减，快被遗忘时主动提醒
- **版本链**（supersede）：新决策覆盖旧决策但留痕，可溯源
- **主动推送**：新消息语义相似度 > 0.78 时自动弹出历史决策卡片

**为什么情景记忆不够用、必须单独建这一层**：


| **维度** | **情景记忆（Episodic）**    | **决策记忆（Decision Layer）**                                         |
| ------ | --------------------- | ---------------------------------------------------------------- |
| 存什么    | "上次说了什么"（事件流）         | "为什么这样决定"（逻辑结论）                                                  |
| 结构化程度  | 半结构化（fact/goal 等宽泛类型） | 强结构化（statement + reasons + objections + alternatives + deadline） |
| 版本管理   | 无，按时间追加               | 有 supersede 链，新覆盖旧                                               |
| 遗忘机制   | 无（历史永久保留）             | 有艾宾浩斯衰减 + 复习提醒                                                   |
| 主动推送   | 无（只在 prompt 里被动返回）    | 有（实时 cosine 检测，超阈值弹卡片）                                           |
| 团队可见   | 个人 session 级          | 项目/群聊级，多人共享                                                      |


**最核心的理由**：情景记忆存的是"发生了什么"，决策记忆存的是"为什么这样做"。企业场景里真正反复被需要的是后者——"我们当时为什么否了方案 A" 这类信息，情景记忆里找不到，因为当时聊天的原文可能横跨几十条消息，被压缩提炼时细节早就丢了。

---

### **记忆聚合路由（HiarchMemoryStore）**

这不是单独一种记忆，而是上面所有层的"调度器"。每次 Agent 收到消息时，`aggregation_memory(current_message)` 这一个入口决定"这条消息去哪个记忆层取上下文"。

新增的 Router 逻辑：

- 消息语义命中某个历史决策的 topic → 走 Decision Layer
- 否则 → 走 Episodic（LightRAG hybrid 检索）
- 两路结果可以叠加返回，互不排斥

**六层记忆的生命周期对比**：

消息发生

   │

   ├─→ STM（立即放入，上限 1000 条）

   │        │ 满了

   │        ▼

   │   Episodic（压缩提炼，长期保留）

   │   Decision（结构化抽取，带遗忘曲线）

   │

   ├─→ Working（多步任务时临时用，任务完 → 清）

   │

   └─→ Semantic（外部知识库，手动/批量导入，永久）

读取时（每次对话前）：

   Router → 决策命中 → Decision Layer

          → 否则    → Episodic

          → 叠加    → 合并注入 system prompt
